사내 자료 수집
시험 보고서·IEC/EN 규격·설계 가이드·BOM·DAQ 로그 등 다양한 형식 통합
PDF·DOCX·XLSX·CSV·HTML
A1 · INGEST
사내 시스템·파일 서버·DAQ 데이터에서 원본 자료를 자동 수집. 각 형식별 파서 필요.
입력: 다중 소스 (10+ 시스템)
도구: Apache Tika, unstructured.io, pdfplumber
출력: 텍스트 + 메타데이터 (출처·작성일·작성자·도메인)
주기: 사전 1회 + 신규 자료 발생 시 증분 (Airflow/Prefect)
이슈: PDF 표·이미지 OCR (Tesseract/Donut), 단위 통일 (mm/inch, kg/lb), 한국어·영어 혼용
A2 · CHUNK
청크 분할 (Chunking)
긴 문서를 검색 단위로 절단. 너무 크면 정확도↓, 너무 작으면 맥락↓
512~1024 토큰 / 50~100 오버랩
A2 · CHUNK
한 문서를 검색 단위(청크)로 분할. 크기·중첩·구조 인식 분할이 RAG 정확도의 핵심 변수.
크기: 512~1024 token (한국어 ≈ 800~1500자)
오버랩: 50~100 token (맥락 유지)
전략:
・Recursive Character (LangChain 기본)
・Sentence-aware (kss·KoNLPy)
・Semantic Chunking (의미 경계)
・Structure-aware (§·표·헤더 보존)
이슈: 시험 보고서의 표 데이터는 분할 시 의미 손실 → 표 별도 처리 필요
A3 · EMBED
임베딩 (Embedding)
청크 → 의미 좌표(고차원 벡터). "의미가 가까우면 좌표도 가깝다"
1024~3072차원 float vector
A3 · EMBED
청크를 고정 차원의 실수 벡터로 변환. 학습된 모델이 의미가 비슷한 텍스트는 가까운 좌표로 매핑.
모델 후보:
・voyage-3 (1024차원, Anthropic 추천)
・text-embedding-3-large (3072d, OpenAI)
・BGE-M3 (1024d, 다국어, 로컬)
・KURE-v1 (1024d, 한국어 특화)
차원 trade-off: 1024d (속도·메모리 균형) vs 3072d (정확도↑ 비용·저장공간↑)
호출 수: 청크 수만큼 (10만 청크 ≈ $5~15)
한국어: KURE > BGE-M3 > voyage > OpenAI 순 (벤치마크)
A4 · INDEX
벡터 DB 적재
벡터 + 메타데이터를 검색 최적화 자료구조로 저장. ANN 인덱스 구축
HNSW · IVF-PQ · DiskANN
A4 · INDEX
벡터를 그대로 두면 100만 건 검색에 수 초 걸림 → 근사 최근접 이웃(ANN) 인덱스로 ms 수준 검색.
알고리즘:
・HNSW — 그래프 기반, 메모리 多, 정확도 高 (기본)
・IVF-PQ — 군집+양자화, 메모리 少, 정확도 中
・DiskANN — SSD 기반 대용량
HNSW 파라미터: M=16, efConstruction=200, efSearch=100
벡터 DB 후보:
・Qdrant (Rust, self-hosted, 추천)
・Milvus (대규모)
・pgvector (PostgreSQL 확장, ⓑ DB와 통합)
・Chroma (개발용)
저장 용량: 1024d × float32 = 4KB/벡터 → 10만 청크 ≈ 400MB

B1 · QUERY
설계자의 자연어 질문을 받음. ① Interpreter 단계에서 의도·모델·키워드를 추출해 검색에 최적화된 쿼리로 변환.

B2 · EMBED
쿼리 임베딩
A3과 동일 모델로 질문을 같은 좌표 공간에 매핑
1024d query vector
B2 · EMBED (쿼리)
사용자 질문을 A3에서 사용한 동일 임베딩 모델로 벡터화. 같은 모델 사용이 필수 — 다른 모델 쓰면 좌표계가 달라 검색 불가.
입력: "SMER 저하 응축기 UA HPWD-24-EU"
출력: vec[1024], float32
지연시간: 50~150ms (API)
최적화: 쿼리 캐싱 (자주 묻는 질문 30% 캐시 시 비용 30%↓)
B3 · SEARCH
유사도 검색 (Top-K)
쿼리 벡터와 가장 가까운 K개 청크 검색. 코사인 유사도 기준
K=10~20 · cosine sim
B3 · SEARCH
코사인 유사도로 가장 가까운 K개 청크 추출. ANN 인덱스 사용으로 ms 수준 검색.
K (Top-K): 10~20 (Reranking 전), 3~5 (LLM 입력)
유사도: cosine = (q·d) / (‖q‖‖d‖) ∈ [-1, 1]
임계값: 0.65 이상만 채택 (도메인 무관 결과 차단)
지연시간: 5~30ms (HNSW, 100만 벡터)
하이브리드 검색: 의미(vector) + 키워드(BM25) 결합
・RRF (Reciprocal Rank Fusion) 또는
・weighted_score = α·vector + (1-α)·BM25 (α=0.6 기본)
필터링: payload 조건 (예: dept="R&D 1팀", date>2024-01)
B4 · RERANK
재랭킹 (Reranking)
Top-K 결과를 더 정확한 모델로 재정렬. 정확도↑ (선택적)
Cross-encoder · K→K'
B4 · RERANK
B3은 빠르지만 부정확 → 더 정확한 Cross-encoder로 Top-K 결과만 재정렬. 정확도 10~20%↑.
Bi-encoder vs Cross-encoder:
・Bi (B3): query·doc 따로 임베딩 → 빠름 (ms)
・Cross (B4): query+doc 함께 입력 → 정확 (100ms)
모델 후보:
・cohere-rerank-3 (API)
・BGE-Reranker-v2 (오픈소스)
・jina-reranker-v2 (다국어)
흐름: 20개 Bi 검색 → 5개 Cross rerank → LLM 입력
비용: 쿼리당 +$0.001
발표 1차에는 생략 가능 (Phase 1 후반에 도입)
B5 · GENERATE
LLM 답변 생성
검색된 청크 + 원본 질문을 외부 LLM API에 전달 → 근거 기반 답변
Top-K → context → answer
B5 · GENERATE
검색된 청크들을 컨텍스트로 LLM에 전달. "이 자료를 근거로만 답하라" 시스템 프롬프트가 환각 방지의 핵심.
프롬프트 구조:

비용: Sonnet 4.5 기준 $0.02~0.05/회

・citations (근거 자동 인용)
・structured output (JSON 강제)
・tool-use (DB·시뮬 추가 호출)

발표 시 청중이 자주 묻는 4가지
CHUNKING
청크 분할
긴 문서 → 검색 단위로 절단. 크기·오버랩·구조 인식이 정확도의 핵심 변수.
CHUNKING
긴 문서(수십~수백 페이지)를 그대로 임베딩하면 의미가 평균화돼 검색이 부정확. 의미 단위로 분할이 핵심.
주요 파라미터:
・chunk_size: 512~1024 token
・chunk_overlap: 50~100 token
・separator: 단락·문장·문자 우선순위
고급 전략:
・Semantic Chunking (의미 경계 자동 검출)
・Late Chunking (문서 전체 임베딩 후 분할)
・Document-aware (시험 보고서·규격 §·표 별도)
평가: Precision@K, Recall@K로 청크 전략 비교
EMBEDDING
임베딩
텍스트 → 고차원 벡터 (1024~3072d). 학습된 모델이 의미를 좌표로 매핑.
EMBEDDING
언어 모델의 최종 hidden state를 추출 → 의미 좌표로 사용. 비슷한 의미는 가까운 좌표에 위치.
모델 유형:
・Bi-encoder (RAG 표준): 빠름, 정확도 中
・Cross-encoder (Reranker): 느림, 정확도 高
차원:
・1024d (voyage-3, BGE-M3) — 균형
・3072d (text-embedding-3-large) — 정확도↑
・Matryoshka (256~3072 가변) — 동적 trade-off
한국어 성능:
・KURE-v1 > BGE-M3 > voyage-3 > OpenAI
(KLUE-MRC 벤치마크 기준)
주의: A3·B2는 반드시 같은 모델
COSINE SIM
코사인 유사도
두 벡터 사이 각도로 의미 유사도 계산. -1(반대) ~ +1(동일).
COSINE SIMILARITY
두 벡터의 방향이 얼마나 같은가. 길이는 무시 (방향만 비교) → 임베딩 비교의 표준.
수식: cos(θ) = (A·B) / (‖A‖‖B‖)
범위: -1 ≤ cos ≤ +1
・1.0: 의미 완전 동일
・0.0: 의미 무관 (직교)
・-1.0: 의미 반대 (실제로 거의 없음)
RAG 임계값:
・> 0.85: 거의 같은 내용
・0.65~0.85: 관련 있음 (채택)
・< 0.65: 무관 (제외)
대안: dot product, Euclidean (정규화하면 등가)
구현: 임베딩 정규화 (unit vector) → cos = dot product
RERANKING
재랭킹
Top-K → Cross-encoder로 재정렬. 정확도 10~20%↑, 비용·지연시간↑.
RERANKING
검색 1차 결과(20개)를 더 정확하지만 느린 모델로 재정렬 → 상위 5개만 LLM에. 정확도-속도 trade-off의 표준 해법.
왜 필요한가:
・Bi-encoder: query·doc 따로 임베딩 (빠름·부정확)
・Cross-encoder: query+doc 함께 처리 (정확)
・모든 문서를 Cross로 처리하면 너무 느림 → 후보 좁힌 후 적용
지표 개선 (벤치마크):
・nDCG@10: +12~18%
・MRR: +15~22%
비용·지연:
・+$0.0005~0.001/쿼리
・+50~200ms (20개 재랭크 기준)
모델 선택:
・cohere-rerank-3 (API, 다국어)
・jina-reranker-v2 (한국어 OK)
・BGE-Reranker-v2-m3 (오픈소스)
도입 시점: Phase 1 후반 (M5~M7)
